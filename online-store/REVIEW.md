# Code Review Report: Online Store

## 1. Архитектура и Паттерны (Architecture & Design Patterns)
- **Бизнес-логика размыта:** Часть логики находится в базе данных (хранимые процедуры `createOrder`, триггеры, функции), а часть в `main.cpp` (проверка наличия товара `if (it->stock < qty)`, расчет суммы заказа на клиенте). Это нарушает принцип единственной ответственности (Single Responsibility Principle) и приводит к дублированию логики (база данных также вычисляет итоговую сумму заказа и проверяет остатки).
- **Паттерн Strategy:** Реализован для выбора способа оплаты (`PaymentStrategy`), что является хорошим решением. Однако логика оплаты никак не связана с самим заказом (сумма просто передается в метод).
- **Отсутствие паттерна Repository:** Сервисы `StoreService` и `ReportService` напрямую работают с SQL-запросами в виде строк. Рекомендуется выделить слой доступа к данным (Data Access Layer, например паттерн Repository), чтобы изолировать SQL-запросы от бизнес-логики.
- **Anemic Domain Model (Анемичная модель предметной области):** Классы `User`, `Order`, `Product` практически не содержат поведения, представляя собой лишь структуры данных. Вся логика вынесена в сервисы и `main.cpp`. В классе `User` заявлена агрегация (`orders_`), но по факту в коде заказы туда не добавляются.

## 2. Безопасность (Security)
- **SQL-инъекции:** В проекте используется прямое форматирование строк и конкатенация для формирования SQL-запросов (например, `"WHERE user_id = " + std::to_string(userId)`). Хотя `std::to_string` в данном случае защищает от классических инъекций, такой подход считается антипаттерном. Необходимо использовать параметризованные запросы (`prepared statements`), которые предоставляет `libpqxx`.
- **Хардкод учетных данных:** В `main.cpp` захардкожена строка подключения к базе данных с паролем (`"host=localhost ... password=stef4587"`). Это грубейшее нарушение безопасности. Строка подключения должна передаваться через переменные окружения или конфигурационный файл.

## 3. База данных и конкурентность (Database & Concurrency)
- **Состояние гонки (Race Condition) при создании заказа:** В `StoreService::createOrder` вызывается хранимая процедура `CALL createOrder(...)`, а затем выполняется `SELECT order_id FROM orders WHERE user_id = ... ORDER BY order_id DESC LIMIT 1;`. В высоконагруженной системе между `CALL` и `SELECT` может вклиниться другой запрос, и метод вернет чужой ID заказа. Хранимая процедура должна возвращать `order_id` (например, использовать функцию вместо процедуры, возвращающую `INT`, или параметр `INOUT`), который затем читается на клиенте.
- **Обработка транзакций:** В `DatabaseConnection` есть методы для работы с транзакциями, но они не используются. Методы `executeQuery` и `executeNonQuery` создают собственные локальные транзакции (объект `pqxx::work`) на каждый запрос. Это значит, что если нужно выполнить несколько связанных запросов атомарно из C++, это будет невозможно без изменения кода сервисов.

## 4. Ошибки проектирования C++ и Производительность (C++ Practices & Performance)
- **Неэффективная передача и обработка данных:** Метод `DatabaseConnection::executeQuery` возвращает `std::vector<std::vector<std::string>>`. Это приводит к множественным аллокациям памяти и постоянной конвертации типов туда-обратно (например, числа из БД конвертируются в строки, а в `StoreService::getProducts` строки обратно парсятся в `int` и `double` с помощью `std::stoi` и `std::stod`). Лучше возвращать `pqxx::result` или мапить данные непосредственно в структуры (ORM/Mapper).
- **Обработка ошибок:** В случае ошибки в `std::stoi` (если, например, в БД окажется NULL, хотя схема защищена `NOT NULL`) программа упадет с исключением `std::invalid_argument`. Нет обертки вокруг `libpqxx` исключений для разделения уровней абстракции БД и сервисов.
- **Умные указатели:** Использование `std::make_unique` и `std::shared_ptr` — отличная практика (например, в фабрике пользователей в `login()`), что позволяет избежать утечек памяти.
- **Алгоритмы STL:** Хорошее использование `std::copy_if`, `std::find_if`, `std::accumulate` в связке с лямбдами, но `std::copy_if` для фильтрации запасов делает копию. Если товаров много, это неэффективно (особенно при выводе). Лучше фильтровать на уровне SQL-запроса (`WHERE stock_quantity > 0`).

## 5. Прочие замечания
- **Создание директорий:** Использование `std::filesystem::create_directories` в `ReportService` — отличное современное решение (C++17) для автоматического создания структуры каталогов под отчеты.
- **Форматирование CSV:** Ручная реализация `csvEscape` работает, но лучше использовать готовую библиотеку для работы с CSV, чтобы избежать краевых случаев с кодировками или спецсимволами.
- **Тестирование:** В проекте нет модульных или интеграционных тестов, что делает рефакторинг текущей архитектуры рискованным.

## Итог
Проект хорошо демонстрирует базовые возможности интеграции C++ с PostgreSQL (libpqxx), использование умных указателей и полиморфизма.
**Приоритетные направления для рефакторинга:**
1. Использование параметризованных запросов для защиты от SQL-инъекций.
2. Вынос строки подключения в переменные окружения.
3. Исправление логики получения ID заказа в `StoreService::createOrder` (возвращать ID напрямую из SQL функции/процедуры), чтобы избежать Race Condition.
4. Вынос логики доступа к данным в отдельный слой (Repository).
